Add simple global access protection to the entire application.

The goal is to protect the app for internal testing without building a full multi-user authentication system yet.

--------------------------------------------------
1. GOAL
--------------------------------------------------

Protect the whole app behind one shared password so that only authorized colleagues can access it.

This is temporary MVP protection and should be implemented in a clean way so it can later be replaced or extended with proper authentication.

--------------------------------------------------
2. ACCESS MODEL
--------------------------------------------------

Implement one global application password.

Behavior:
- Every visitor must pass through a login page before accessing the app
- There is one shared password for the whole application
- After entering the correct password, the user gets access to the app
- After entering a wrong password, access is denied

Do NOT build user accounts, registrations, password reset, or email login.

--------------------------------------------------
3. PASSWORD STORAGE
--------------------------------------------------

The shared password must NOT be hardcoded.

Store it only as an environment variable / secret:
- APP_ACCESS_PASSWORD

Validation must happen only on the server side.

The password must never be exposed to the frontend.

--------------------------------------------------
4. LOGIN FLOW
--------------------------------------------------

Create a login page at:
- /login

Login page requirements:
- simple clean UI
- password input
- submit button
- error message for invalid password

Flow:
1. User opens app
2. If not authenticated, redirect to /login
3. User enters password
4. Server validates password against APP_ACCESS_PASSWORD
5. If valid:
   - create authenticated session
   - redirect to dashboard
6. If invalid:
   - stay on /login
   - show generic error message: "Invalid password"

--------------------------------------------------
5. SESSION HANDLING
--------------------------------------------------

Use a secure session or signed cookie approach.

Requirements:
- use httpOnly cookie
- use secure cookie in production
- use sameSite=lax or stricter
- set reasonable expiration, e.g. 7 days
- allow logout by clearing the session/cookie

Add a logout button in the app layout.

--------------------------------------------------
6. ROUTE PROTECTION
--------------------------------------------------

Protect all application routes except:
- /login
- static assets if needed
- authentication endpoint if separate

All other routes must require an active authenticated session.

If no valid session exists:
- redirect to /login

This includes at minimum:
- /
- /dashboard
- /timesheet
- /clients
- /projects
- /employees
- /holidays
- /reports

--------------------------------------------------
7. SERVER REQUIREMENTS
--------------------------------------------------

Implement authentication check on the server side or middleware layer.

Do NOT rely on frontend-only protection.

If an unauthenticated user manually enters a protected URL:
- they must still be redirected to /login

--------------------------------------------------
8. LOGIN SECURITY
--------------------------------------------------

Add basic brute-force protection.

Requirements:
- simple rate limiting on login attempts
- example:
  - after too many failed attempts from same IP in short time, temporarily block or delay further attempts

Do not over-engineer this, but add a basic safeguard.

--------------------------------------------------
9. UI REQUIREMENTS
--------------------------------------------------

Login page should be minimal and professional.

Include:
- app name
- short message like "Internal access only"
- password field
- submit button

Do NOT include:
- sign up
- forgot password
- username/email field

--------------------------------------------------
10. ERROR HANDLING
--------------------------------------------------

Use generic errors only.

For invalid password:
- show "Invalid password"

Do not reveal:
- whether secret is missing
- server internals
- stack traces

If APP_ACCESS_PASSWORD is missing in environment:
- fail safely on server startup or show a clear developer error in logs
- do not expose this to end users

--------------------------------------------------
11. IMPLEMENTATION NOTES
--------------------------------------------------

Preferred implementation:
- login POST endpoint validates password server-side
- on success, create session
- middleware protects app routes
- logout endpoint destroys session

Keep implementation simple and maintainable.

--------------------------------------------------
12. OUT OF SCOPE
--------------------------------------------------

Do NOT build:
- user-specific authentication
- role-based access
- employee-specific login
- password reset
- email verification
- SSO
- OAuth

This task is only about global app protection.

--------------------------------------------------
13. ACCEPTANCE CRITERIA
--------------------------------------------------

The feature is complete when:

1. Opening the app without authentication redirects to /login
2. Entering the correct shared password grants access
3. Entering a wrong password does not grant access
4. Protected routes cannot be accessed directly without authentication
5. Password is stored only in environment variable
6. Session persists across page navigation
7. Logout clears session and redirects to /login
