In resourceBookings.ts, I need to update the booking notification logic 
for CREATE, UPDATE, and DELETE operations.

CONTEXT:
There is a notification_queue table in the existing Neon Postgres database 
with this structure:
  - id SERIAL PRIMARY KEY
  - employee_email TEXT NOT NULL
  - employee_name TEXT NOT NULL
  - project_name TEXT NOT NULL
  - role_name TEXT NOT NULL
  - start_date DATE
  - end_date DATE
  - hours_per_day INTEGER
  - queued_at TIMESTAMPTZ DEFAULT NOW()
  - send_after TIMESTAMPTZ DEFAULT NOW() + INTERVAL '30 minutes'
  - sent BOOLEAN DEFAULT FALSE
  - CONSTRAINT unique_employee_project UNIQUE (employee_email, project_name)

The table does NOT exist yet — create it as part of this implementation 
using a CREATE TABLE IF NOT EXISTS migration at server startup, 
using the existing db and sql imports.

---

BEHAVIOUR CHANGES NEEDED:

1. BOOKING CREATED (POST /resource-bookings)
After a booking is successfully saved to the database:
- Remove the existing N8N_BOOKING_WEBHOOK_URL HTTP POST call entirely
- Instead, upsert into notification_queue:

INSERT INTO notification_queue 
  (employee_email, employee_name, project_name, role_name, 
   start_date, end_date, hours_per_day, send_after)
VALUES (
  [employee email],
  [employee name],
  [project name],
  [role name],
  [start date],
  [end date],
  [hours per day],
  NOW() + INTERVAL '30 minutes'
)
ON CONFLICT (employee_email, project_name) DO UPDATE SET
  role_name = EXCLUDED.role_name,
  start_date = EXCLUDED.start_date,
  end_date = EXCLUDED.end_date,
  hours_per_day = EXCLUDED.hours_per_day,
  send_after = NOW() + INTERVAL '30 minutes',
  sent = FALSE;

This means:
- If employee is booked on multiple projects, each gets its own queue row
- If admin edits the same employee+project booking multiple times within 
  30 minutes, the timer resets and only ONE notification fires with 
  the latest details

2. BOOKING UPDATED (PUT /resource-bookings/:id)
After a booking is successfully updated in the database:
- Same upsert as above using the updated booking details
- This resets the 30 minute timer so the employee gets 
  the latest version not an outdated one

3. BOOKING DELETED (DELETE /resource-bookings/:id)
After a booking is successfully deleted from the database:
- Delete from notification_queue if not yet sent:

DELETE FROM notification_queue
WHERE employee_email = [employee email of deleted booking]
AND project_name = [project name of deleted booking]
AND sent = FALSE;

This prevents a notification firing for a booking that no longer exists.
- If the row was already sent (sent = TRUE), just log it — 
  cancellation emails will be handled later
