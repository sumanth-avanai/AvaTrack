## Context

This is a UI/UX redesign of the existing **Project Status** area in AvaTrack (React/Vite/Tailwind frontend, Express API, PostgreSQL/Drizzle). It covers two views:

1. The **Project Status overview** (portfolio table of all projects)
2. The **Project Status detail view** (single project)

**CRITICAL: Do not modify, delete, or migrate any existing data.** This is a UI/UX and read-path change. Any new fields listed under "New data required" must be added as new, nullable/optional columns with sensible defaults — never by altering or backfilling existing columns in a way that changes current values. If a migration is required, it must be additive only.

---

## Design system constraints (must be followed)

These are established, existing patterns in AvaTrack — do not invent new conventions that conflict with them:

- **Language**: all UI copy is English, regardless of the language used elsewhere.
- **Color coding**: each project already has a `colour` field. Use it directly for the colored dot next to project names. Do not introduce a separate color dimension for client or health status — health uses semantic status colors (success/warning/danger), not the project's color.
- **Labels**: show full text by default, no truncation. Only use truncation + tooltip as a last resort if a column is genuinely too narrow.
- **Filtering**: replace any standalone dropdown filters with a single searchable, groupable filter panel with an active-count badge, matching the existing Resource Planner filter panel component. Unchecking an item hides matching rows (not just dims them).
- **Empty states**: rows/fields with no data stay visible with a clear, subtle label (e.g. "Not assessed", "No hours yet", "No comment") — never a bare "—" and never hidden.

## Part 1 — Project Status overview (portfolio table)

### Layout
- Page header "Project status" with:
  - A single search input ("Search projects...")
  - One "Filter" button (funnel icon) opening the shared filter panel component, grouped by customer, with an active-count badge (e.g. "3")
- **KPI strip** above the table, 4 metric cards:
  - "Active projects" — count of non-completed projects
  - "Needs attention" — count of projects where risk level is High, or budget alert is active (see below), colored with the danger background
  - "Budget alerts" — count of projects at or above the budget threshold (see Part 2 for threshold logic), colored with the warning background
  - "Update overdue" — count of projects whose last status update is older than the update cadence (see "New data required")
- **Quick filter chips** below the KPI strip: "All", "Needs attention", "Update overdue" — clicking filters the table client-side.
- **Table columns**: Project (color dot + name), Customer, PM, Health, Budget, Updated.
  - Remove the "Last Comment" column from the table entirely (it's always empty at this list level — the latest comment now lives in the detail view's "Last comment" metric).
  - **Health cell**: a single risk badge (colored by severity: Low/Medium/High/Not assessed) plus a small trend icon (arrow-up = worsening, arrow-right = stable, arrow-down = improving) computed by comparing the two most recent history entries' risk level. If there's only one history entry (or none), show no trend arrow.
  - **Budget cell**: a horizontal progress bar + percentage, colored semantically: green under 70%, amber 70–90%, red at/above 90%. If no hours are logged yet, show an empty track with the label "No hours yet" instead of a 0%-width bar.
  - **Updated cell**: the date of the last status update. If the project has never had a status update, show "Never" in the warning color. If the last update is older than the cadence threshold, show a small clock icon next to the date in the warning color.
  - Rows where the project needs attention (High risk, or budget alert, or update overdue) get a 3px left accent border in the corresponding severity color (danger or warning), so they visually stand out when scanning the list. Default sort keeps these at the top.
- Keep the existing "Show N completed projects" collapsible section and the "X active, Y completed" summary line below the table, unchanged in behavior.

## Part 2 — Project Status detail view

### Header
- Colored dot (project's `colour`) + project name, customer name, PM name — unchanged from current.
- Keep "Open in Billing" and "Open in Reports" links in the top-right, unchanged.

### Health row
- Three badges: General status (plain colored badge, e.g. "In progress"), Risk level (colored badge + trend arrow, same logic as the overview), Satisfaction/sentiment (colored badge, or "Sentiment not set" if null).

### Key facts panel
Three small cards in a row:
- **Runtime**: project start – end date.
- **Next update due**: computed from the last update date + cadence (see "New data required"). Show in the warning color if due within 3 days or overdue.
- **Last comment**: date of the most recent history entry that has a non-empty comment. If none exists, show "No comments yet".

### Budget breakdown
A horizontal segmented bar plus a 3-column summary grid, showing:
- **Invoiced** (amount + % of total budget) — accent color
- **Logged, not invoiced** (amount + % of total budget) — warning color
- **Remaining** (amount + % of total budget) — neutral/muted, outlined
- A vertical threshold marker on the bar at 90% of total budget (ASSUMPTION: threshold applies to Invoiced + Logged combined — flag this in a code comment so it's easy to change if the actual intended base is "Invoiced only").
- Below the bar: "€X of €Y" total and "90% threshold" label.

### Budget over time
A stacked area ("burn-up") chart:
- X-axis: months from project start to current month.
- Two stacked areas per month: Invoiced (bottom, accent color) and Logged-not-invoiced (top, warning color), both cumulative.
- A dashed horizontal reference line at the total budget amount, and a second dashed line at the 90% threshold.
- A vertical "Today" divider.
- **If — and only if — there are future Resource Planner bookings for this project**, extend the stacked areas past "Today" using projected values derived from those bookings (rendered with a diagonal hatch pattern at reduced opacity to visually distinguish "planned" from "actual", with a legend entry "Planned (bookings)").
- **If there are no future bookings, the chart simply ends at "Today"** — do not extrapolate or guess a trend line. This is a deliberate product decision: an unplanned future must not look like a forecast.
- ASSUMPTION: this chart requires monthly historical values for both Logged and Invoiced amounts. Logged monthly history is confirmed to exist for all projects. Invoiced monthly history is NOT confirmed — if it turns out only a current snapshot exists for Invoiced (no monthly breakdown), render the Invoiced area only from "today" backwards using the current snapshot value flatly, and add a small note "Historical invoiced breakdown not available" under the chart. Do not fabricate historical Invoiced values.

### Next steps
- A simple checklist (2–5 items), each with a checkbox and text, strikethrough + muted color when checked.
- NEW DATA REQUIRED: this needs a new field to persist next-step items per project (e.g. a JSON array of `{ id, text, done }`, or a small related table if you prefer relational modeling — your choice, but keep it simple since this is a lightweight checklist, not a task management system). Add as a new nullable field/table; do not touch existing project fields.

### History
- Vertical timeline (dot + connecting line), one entry per status update, newest first.
- Per entry, two-column layout:
  - **Left column (fixed ~110px width)**: the general status as plain text (not a colored badge — it's contextual, not a severity signal), followed by two small circular icon chips below it:
    - Risk level: shield icon, colored by severity (success/warning/danger), tooltip shows the exact label (e.g. "Medium risk").
    - Satisfaction: mood icon (smile/neutral/frown), colored by severity, tooltip shows the exact label.
  - **Right column**: the comment text if present, or the muted label "No comment" if not — never blank space.

## New data required (flag clearly in your implementation, don't silently assume)

1. **Update cadence**: a way to compute "next update due". If there's no per-project cadence field yet, use a global default of 14 days and add a code comment noting this should become configurable later (either per-project field or app-wide setting).
2. **Next steps checklist**: new field/table as described above.
3. **Monthly Invoiced history**: confirm during implementation whether this exists; if not, follow the fallback described in "Budget over time" above.
4. **Budget threshold base**: confirm whether the 90% threshold should apply to Invoiced+Logged combined (current assumption) or Invoiced only, and make this easy to change (e.g. a single named constant/config value, not hardcoded in multiple places).

## Explicitly out of scope for this iteration

- **Budget breakdown by role** (Senior Consultant / Consultant / etc. with per-role day/rate caps). This depends on a project budget-type distinction (Time & Material vs. other types) that doesn't exist yet in the data model. Do not attempt this now — leave a comment/TODO in the code referencing this for a future iteration.
- Making the "Budget over time" chart filterable by role — same dependency as above.
