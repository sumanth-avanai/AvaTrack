You are helping me extend an existing full‑stack app in this Replit project called **AvaTrack**. AvaTrack is a resource planning, reporting and billing tool.

Your task: Implement a new **“Project Status” (Project Health) module** as a separate, modular feature with its own main navigation entry. You must follow the existing tech stack, patterns and folder structure of this project (frontend, backend, database). Do NOT introduce a completely new framework or stack; infer and reuse what’s already there.

## High-level goal

Create a central place where I can see and maintain the *health* of each project:
- General project status
- Budget status
- Risk level
- Comment + history of changes
- Filters by project, customer and PM

This should work as a standalone module now, but be clean and modular enough to plug into future permissions/roles later.

## Functional requirements (V1)

### 1. New main navigation entry

- Add a new top-level menu item, e.g.:
  - `Resource Planner | Reports | Billing | Project Status | Settings`
- Route for the overview page, e.g.: `/project-status`
- Route for a detail page, e.g.: `/project-status/:projectId`

---

### 2. Project Status Overview (table view)

This is the main overview of all projects.

**Filters:**
- Text search for **Project name** (case-insensitive, contains).
- Dropdown filter for **Customer**.
- Dropdown filter for **PM** (“Project Manager” / person responsible for Project Health).

**Table: one row per project, with at least these columns:**
1. Project Name
2. Customer
3. PM
4. General Status (display as a colored badge)
5. Budget Status (badge)
6. Risk Level (badge)
7. Last Update (timestamp of the last health entry for that project)
8. Last Comment (truncated, e.g. first 80–100 chars, with tooltip for full text)

**Behavior:**
- Applying filters narrows the table (filters can be combined).
- Clicking a row (or a “View details” action) opens the Project Detail / Health view for that project at `/project-status/:projectId`.

**V1 simplifications:**
- No filter by Status or Risk yet.
- No export (CSV/Excel) yet.
- No upsell-/tags-related columns.

---

### 3. Project Detail / Health Card View

The detail view shows the health of a single project.

**Top area (header):**
- Project name
- Customer
- PM
- Optional: buttons/links to open the existing Billing/Reports views for this project (if easily possible, e.g. “Open in Billing” / “Open in Reports”). These can just route to existing pages with the project pre-filtered.

**“Project Health” Card (at-a-glance):**
- Shows the **current** values (the latest health entry) for:
  - General Status (dropdown with badge)
  - Budget Status (dropdown with badge)
  - Risk Level (dropdown with badge)
  - Latest comment (short text, from the latest history entry)
- Editing pattern:
  - Either:
    - Direct inline editing in the card (drop-downs + comment field), OR
    - An “Add Health Update” / “Update Status” button that opens a small form.
  - Choose the approach that best matches the existing UI patterns in this project.

**History section (below or to the side):**
- A list (or timeline) of all past “health updates” for this project.
- Sorted newest first.
- Each entry contains:
  - Timestamp
  - General Status
  - Budget Status
  - Risk Level
  - Comment (full text)

**Behavior when adding an update:**
- The user can set:
  - General Status (pre-filled with current value)
  - Budget Status (pre-filled with current value)
  - Risk Level (pre-filled with current value)
  - Comment (free text)
- On save:
  - Create a new health history entry (see data model below).
  - Update the project’s current status fields (general_status, budget_status, risk_level) to these new values.
  - Refresh the card and the history list.

**V1 requirement:**
- Changes are **append-only**:
  - No editing or deleting history entries in V1.

### Budget Status (budget health)
- `on_track`
- `at_risk`
- `over_budget`

Again, show readable labels and sensible colors (green/yellow/red/grey, etc.).

### Risk Level
- `low`
- `medium`
- `high`

Show as badges with colors (e.g. green, orange, red).

### Comments
- Free-text comment field (string / text) describing what’s going on.
- This comment must be stored as part of each history entry.
- Ideally, when the Budget Status changes, a comment should be required; for other changes it can be optional. If that’s too complex for V1, a simple optional comment field is OK, but still stored in history when provided.

## Data model

Please inspect the existing DB/schema/models and extend them in a way that is consistent with the current project.

Assuming a `projects` table/collection already exists, extend it with:

- `general_status` (string/enum, one of the values above)
- `budget_status` (string/enum)
- `risk_level` (string/enum)
- `customer_id` or `customer_name` (if not already present, so we can filter by customer)
- `project_manager_id` or `pm_name` (if not already present, to filter by PM)

These fields represent the **current** state and should be fast to query for the overview table.

Create a new table/collection, e.g. `project_health_updates` (name can be similar) with at least:

- `id` (primary key)
- `project_id` (FK → projects)
- `timestamp` (datetime)
- `general_status` (string/enum)
- `budget_status` (string/enum)
- `risk_level` (string/enum)
- `comment` (text)
- Optionally `created_by` (if there is already a user model; if not, you can omit or set a default)

**Rules:**
- Whenever a new health update is created:
  - Insert a row into `project_health_updates`.
  - Update the corresponding `projects` row with the new `general_status`, `budget_status`, and `risk_level`.
