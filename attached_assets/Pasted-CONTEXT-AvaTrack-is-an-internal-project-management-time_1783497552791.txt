CONTEXT
AvaTrack is an internal project management / time-tracking tool (React + Vite + 
Tailwind frontend, Express API, PostgreSQL + Drizzle ORM). This task reworks the 
Billing page for a single project.

CRITICAL: Do not alter, migrate, or delete any existing data (time entries, 
projects, roles, employees, existing invoice records) as part of this change. 
Only add new fields/tables as specified below and change the UI/query logic. 
If a migration is required to add new columns, it must be additive and 
backward-compatible with existing rows (use sensible defaults, no data loss).

BACKGROUND / BUG BEING FIXED
Currently all five summary metrics on the Billing page (Budget, Logged, 
Invoiced, Unbilled, Remaining) incorrectly recompute based on the selected 
Period filter. This produces wrong values, e.g. "Invoiced" showing 0 € when 
filtering to "Last Month" even though invoices exist for that project overall.

The fix is to split the page into two independently-scoped sections instead 
of one filter-driven table.

---

SECTION 1 — Project overview (period-independent)

Always reflects the full project lifetime, regardless of any period filter 
elsewhere on the page.

KPI row (4 metric cards):
- Budget = fixed contract value (unchanged, already correct)
- Total invoiced = SUM of amounts across all entries/roles ever marked as 
  invoiced (all time)
- Total logged = SUM of all logged amounts across the full project lifetime 
  (all time)
- Remaining = Budget − Total logged (all time, NOT period-scoped)

Chart (below the KPI row):
- Cumulative line/area chart over the project's full timeline (monthly buckets)
- Three series:
  - Budget: flat dashed reference line at the fixed budget value
  - Logged: cumulative sum of logged amounts, filled area
  - Invoiced: cumulative sum of invoiced amounts, filled area
- X-axis: months from project start to current month
- No interactivity required beyond standard chart tooltips; this is a 
  read-only overview visual

---

SECTION 2 — Ready to invoice (period-scoped)

This section has its OWN period selector (e.g. "June 2026", "May 2026", 
"Custom range"), independent from anything in Section 1. Changing it never 
affects the KPI cards or chart in Section 1.

Table columns: checkbox | Role/Employee | Logged (period) | Unbilled (period) | 
Status

Formulas (period-scoped, i.e. computed only over entries falling inside the 
selected period):
- Logged (period) = SUM of logged amounts for that role/employee within the 
  selected period. This is a historical fact and does not change once the 
  period has passed.
- Unbilled (period) = SUM of logged amounts within the selected period for 
  entries/blocks whose invoiced status is NOT set. This can be less than 
  Logged (period) if some of that period's activity has already been 
  invoiced, and converges toward 0 the more has been billed.
- Status: "Open" if any unbilled amount remains for that row in the period, 
  "Invoiced" if fully covered by a prior invoice action.

ASSUMPTION TO CONFIRM: invoiced status is tracked per role/employee block 
(matching the existing "Mark as" dropdown behavior), not per individual daily 
time entry. This means Unbilled cannot be split within a role/employee row if 
that row has both invoiced and non-invoiced days in the same period — treat 
each role/employee row as fully invoiced or fully open per period. Flag this 
in code review if the existing data model already tracks status at the daily 
entry level — the formulas above would need to be computed at that finer grain 
instead.

Checkbox selection default: pre-select all rows with Status = "Open" in the 
current period; leave "Invoiced" rows unchecked by default.

Below the table: a "Generate invoice" button (enabled only when at least one 
row is checked).

---

SECTION 3 — Generate invoice flow

Clicking "Generate invoice" opens a confirmation modal (not a document 
generator — this only records that an invoice was created elsewhere):

Modal contents:
- Header: "Mark as invoiced"
- Subheader: selected period + count of selected positions
- Summary list: each selected role/employee with its amount, plus a total
- Optional text input: "Invoice reference (optional)" — free text, e.g. 
  "INV-2026-0143"
- Cancel / Confirm buttons

On Confirm:
- Set invoiced status = true for all selected role/employee blocks within the 
  selected period
- Create a new entry in Invoice history with: date created (now), period 
  covered, total amount, list of included role/employee positions, and the 
  optional reference string (nullable)
- Close modal, refresh Section 2 table (selected rows now show Status = 
  "Invoiced" and Unbilled = 0 for that period) and Section 1 KPIs/chart 
  (Total invoiced increases accordingly)

No actual PDF/document is generated by this action — this is intentionally 
out of scope for now and may be added later as a separate feature.

  entity if it doesn't already exist at that granularity (see ASSUMPTION

- Invoice history entries need: id, project_id, created_at, period_start,

ACCEPTANCE CRITERIA
[ ] Budget, Total invoiced, and Remaining in Section 1 do NOT change when 
    Section 2's period filter is changed
[ ] Total invoiced in Section 1 equals the sum of all Invoice history entries 
    for the project, all time
[ ] Section 1 chart renders three series (Budget flat line, Logged cumulative 
    area, Invoiced cumulative area) across the project's monthly timeline
[ ] Section 2's Logged (period) reflects only entries within the selected 
    period and does not change after the period has passed
[ ] Section 2's Unbilled (period) drops to 0 for a role/employee once that 
    row has been included in a confirmed invoice for that period
[ ] Rows already fully invoiced for the selected period default to unchecked; 
    open rows default to checked
[ ] "Generate invoice" button is disabled when no rows are selected
[ ] Confirmation modal shows correct summary, total, and accepts an optional 
    free-text reference
[ ] Confirming the modal updates status in Section 2, adds an entry to 
    Invoice history, and updates Section 1 KPIs — without a page reload 
    losing state
[ ] No existing time entries, projects, roles, employees, or invoice history 
    records are modified, deleted, or reformatted by this change
[ ] All UI copy is in English, sentence case, no exclamation marks
