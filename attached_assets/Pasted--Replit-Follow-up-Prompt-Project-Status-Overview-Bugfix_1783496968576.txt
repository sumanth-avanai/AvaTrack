## Part 1 — Bugfixes

### 1.1 "Needs attention" and "Update overdue" KPI counts are wrong
In the current implementation, both KPIs show the same count as total active projects (i.e. every single project is flagged), which makes them useless as a signal. Investigate and fix:
- Confirm the "today" reference date used in the overdue calculation is the actual current date, not a stale or hardcoded test date.
- Confirm the update cadence default (14 days) is being compared correctly against each project's last status-update timestamp, not against some other date field.
- Confirm "Needs attention" only counts projects where risk level is **High**, or the project is at/above the budget threshold — not every project with any non-empty health field.
- After the fix, verify manually against a few real projects that the counts make sense (i.e. most healthy, recently-updated projects should NOT appear in either count).

### 1.2 Health column empty state shows a bare "—"
For projects with no risk/status assessment yet, the Health column currently renders a plain "—". Replace this with a visible "Not assessed" badge, styled muted/neutral (same treatment as other empty states in the app, e.g. the Budget column's "No hours yet").

### 1.3 Left accent border uses the wrong color source
The left accent border on table rows currently appears to use each project's own `colour` field (the same color as the project dot), so it shows on every row regardless of status. It should instead:
- Only appear on rows that are flagged (High risk, budget alert, or update overdue).
- Use a severity color (danger red for High risk / budget alert, warning amber for update-overdue-only), not the project's own color.
- Rows with no flags get no left border at all.
This must not conflict with or replace the existing colored project dot next to the project name — those are two different signals (project identity vs. attention needed).

### 1.4 Trend arrow shows "—" instead of no icon
When there's insufficient history (0 or 1 status update) to compute a risk trend, the Health column currently shows a "—" placeholder next to the risk badge. Change this to render nothing at all in that case — no arrow, no dash, just the risk badge alone.

## Part 2 — Show absolute budget amounts, not just percentage

Currently the Budget column only shows a progress bar and a percentage. Change the cell layout to:
- Progress bar on top (unchanged coloring: green <70%, amber 70–90%, red ≥90%)
- Below the bar, two pieces of text on the same line:
  - Left: absolute amount, formatted like `€91k / €198k` (consumed / total budget)
  - Right: the percentage, colored to match the bar's threshold color
- If no hours are logged yet, keep the existing "No hours yet" label instead of amounts (don't show "€0k / €198k").

This will make each Budget cell slightly taller (bar + one text line instead of bar + inline percentage) — adjust row height/vertical padding accordingly so the table doesn't feel cramped.

## Part 3 — Grouping by Customer or PM

### Control
Add a "Group by" control next to the existing search/filter row: a small segmented control with three options — **None** (default), **Customer**, **PM**.

### Behavior when grouped
- Rows are grouped into sections by the chosen field, each with a **group header row**:
  - Group name (bold)
  - Project count in that group (e.g. "3 projects")
  - If any project in the group is flagged as needing attention, show a small "X needs attention" indicator on the right side of the header, in the warning/danger color as appropriate
  - A collapse/expand chevron; groups default to expanded
- When grouped by Customer, hide the Customer column from the table (it's redundant with the group header). Same logic for PM: when grouped by PM, hide the PM column.
- **Projects with no value for the grouping field** (e.g. no PM assigned, when grouping by PM) are NOT placed into their own group (do not create a "No PM assigned" or "Unassigned" group). Instead, append them as plain rows at the very bottom of the table, after all named groups, with no group header wrapping them.
- Search, the filter panel, and the existing quick filter chips ("All" / "Needs attention" / "Update overdue") must keep working together with grouping: filtering happens first, then the filtered result set is grouped.
- Sort order within each group stays the same as the current default sort (flagged/attention-needed rows first).

### Persistence
There are no user accounts in AvaTrack yet (single shared admin password only) — so do **not** build server-side per-user preference storage for this. Instead, persist the following in the browser's `localStorage`:
- The selected "Group by" mode
- The collapsed/expanded state of each group (keyed by group name)

These should be restored on page load/reload, scoped to the browser, not synced across devices or users.
