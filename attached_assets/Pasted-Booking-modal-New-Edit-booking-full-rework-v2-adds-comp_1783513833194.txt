Booking modal (New/Edit booking) — full rework, v2 (adds compact Edit summary)

CONTEXT
AvaTrack is a React/Vite/Tailwind + Express + PostgreSQL (Drizzle) time-tracking
and resource planning tool. This prompt covers a rework of the "New booking"
modal used from the Resource Planner timeline, plus a set of shared and
Edit-specific changes. All UI copy must be in English. Do not change unrelated
parts of the app.

CRITICAL — DATA SAFETY
- Do not modify, delete, or migrate any existing booking records, project
  records, role records, or absence records as part of this change.
- If the bookings table needs new columns (status, schedule_mode, etc.), add
  them as nullable / with safe defaults so existing rows remain valid.
- Existing bookings created before this change must continue to render
  correctly on the timeline and in the Edit modal.
- Every number currently computed and shown in the Edit modal (working days,
  public holidays subtracted, bookable days, budget consumption, past vs.
  future undelivered/re-plannable amounts, the full Budgeted/Logged/Invoiced/
  Re-plannable/Unplanned/Free breakdown, and the per-employee slot lists)
  must remain computed and accessible after this change — this prompt only
  changes WHERE and HOW that data is displayed (condensed into a bar +
  tooltip + link), not what is calculated or stored.

SCOPE — NEW VS EDIT MODAL
- The schedule mode switcher (Zeitraum/Dauer/Budget, section 4) applies ONLY
  to the "New booking" flow. Once a booking exists, editing it always shows
  the concrete date-range view (current Zeitraum-style start/end date +
  per-day/weekday-template hours grid) — regardless of which mode was
  originally used to create it.
- The following field-level changes apply to BOTH New and Edit:
  - Project field → searchable client-grouped combobox (3b)
  - Status toggle Tentative/Confirmed (3d)
  - Notes: collapsed behind "+ Add note" when empty; pre-expanded if the
    booking already has notes (relevant for Edit).
- The Role field's inline remaining-budget badge (3c) applies to NEW booking
  ONLY. In the Edit modal, this would duplicate the new Role-budget bar
  (section 7) — do not add the badge there, the bar is the single source of
  truth for that context.
- The Assignees multi-person field (3a) is NEW-booking only. The Edit modal
  continues to represent exactly one person per booking, as today.
- Sections 4 (mode switcher + Dauer/Budget modes) are NEW-booking only.
- Section 7 (compact Role-budget bar) is EDIT-booking only — it replaces the
  Edit modal's current three stacked blocks ("This slot", "This slot — past
  vs. future", "Role budget" table) and the two always-expanded slot lists.

Out of scope / explicitly not solved here: fallback behavior for projects
with no roles defined ("No roles defined for this project"). Leave existing
behavior untouched for that case.

---

1. ENTRY POINTS (New booking only)
- Keep the existing per-employee-row "+" button. It opens the New booking
  modal with that employee pre-filled as the first assignee (still editable
  — more people can be added).
- Add a new toolbar-level "New booking" button (top of the Resource Planner
  view). It opens the same modal with an empty Assignees field.

2. MODAL LAYOUT — SECTIONS (New booking)
Replace the current flat list of stacked fields with two visually separated
sections, each with a small muted 12px section label and a thin divider
above "Schedule":
  - "People & project"
  - "Schedule"

3. SECTION: PEOPLE & PROJECT

3a. Assignees (New booking only)
- Multi-select field showing selected people as removable chips (initials
  avatar + name) plus an "Add person" button that opens a searchable people
  picker.
- The Create button label reflects the count: "Create booking" for 1 person,
  "Create N bookings" for multiple. Each person gets an independent booking
  record with identical project/role/schedule/status/notes — editable and
  deletable independently afterward via the Edit modal.

3b. Project field (New AND Edit)
- Replace the current single flat dropdown showing "Project name (Client)"
  with a searchable combobox:
  - Typing filters by both project name and client name.
  - Results are grouped under client-name group headers (non-selectable),
    with matching projects listed underneath, indented.
  - Matched substrings in the result list are rendered in medium weight
    (500) for scannability.
- Reuse the existing client-grouped searchable filter component from the
  Resource Planner timeline filter panel if feasible, rather than building
  a new one.

3c. Role field
- Keep the existing behavior (role name + rate shown in the dropdown, e.g.
  "Trainer — €1.000/day").
- New booking ONLY: add an inline badge to the right of the field showing
  remaining role budget (e.g. "42d left" or in hours if the role's budget is
  tracked in hours). Badge color: success tint under 80% consumed, warning
  tint above 80%, danger tint if already exceeded. Informational only, never
  blocks saving.
- Edit booking: no badge here — remaining budget is shown via the bar in
  section 7 instead.

3d. Status toggle (New AND Edit)
- Two-option segmented control: "Tentative" / "Confirmed". Default for new
  bookings: "Confirmed" (matches current implicit behavior). Editable
  afterward in the Edit modal.
- Tentative bookings do NOT count against the role budget until switched to
  Confirmed. They still render on the timeline, but with a dashed border
  instead of a solid one (visual-only change on the timeline).
- Both the New modal's budget summary text and the Edit modal's budget bar
  (section 7) must reflect a Tentative booking's exclusion from budget
  consumption — confirm exact bucket mapping against the existing Budgeted/
  Invoiced/Re-plannable/Unplanned/Free logic before wiring this up.

4. SECTION: SCHEDULE (New booking only)

4a. Schedule mode switcher
Three-option segmented control at the top of this section, New booking only:
  "Zeitraum" (Date range) / "Dauer" (Duration) / "Budget"
Default: "Zeitraum". Only the fields relevant to the selected mode are shown.

4b. Mode: Zeitraum (Date range) — existing logic, unchanged
- Start date, End date.
- "Set per weekday" toggle + quick-preset buttons (Mo–Fr 8h / Mo–Do 8h /
  Mo–Fr 4h) + per-day hour grid, exactly as currently implemented.
- Per-day grid: any day that is a public holiday or an approved absence for
  the selected assignee(s) is automatically set to 0h, using the existing
  absence visual treatment (neutral gray + icon), remaining manually
  overridable. Add a short inline note listing which day(s) were adjusted,
  e.g. "Do 16.07 is a public holiday — hours set to 0 automatically."
- Note: the current Edit modal's hours grid uses a weekday-template pattern
  (Mo/Di/Mi/Do/Fr without concrete dates), not per-calendar-date. The
  holiday-auto-0h behavior described here applies only where the grid is
  bound to concrete calendar dates (New booking, Zeitraum mode). Do not
  attempt to apply per-date holiday logic to the Edit modal's weekday-
  template grid — leave that grid's current behavior untouched.

4c. Mode: Dauer (Duration) — New booking only
- Fields: Start date, Duration (number input + unit selector: Tage / Wo /
  Mo), Hours/day.
- Unit "Wo" or "Mo": show the weekly pattern picker (4e) instead of a
  per-day grid. End date is calculated and shown read-only.
- Unit "Tage": no weekday pattern picker. Book N consecutive working days
  from the start date, auto-skipping weekends/holidays/absences.
  - If the start date falls on a weekend/holiday, auto-shift forward to the
    next working day and show a non-blocking info banner: "Booked on
    [original date] shifted to [actual date] — start date fell on a
    weekend/holiday, moved to the next working day automatically."

4d. Mode: Budget — New booking only
- Fields: Start date, weekly pattern picker (4e), Target (number + unit,
  freely choosable between "Tage"/"Stunden", not locked to the role's own
  budget unit), optional "Spätestens bis" (cap) date.
- Calculate and display read-only "Calculated end date", skipping weekends/
  holidays/absences, stopping at target or cap date, whichever comes first.
- If a cap date makes the target unreachable, show a non-blocking warning
  (e.g. "Only 32 of 40 days reachable by 15.12.2026") but still allow
  saving as-is.
- Summary: "Calculated end date: [date]" / "Bookable days to target: Xd (Y
  holidays skipped)".

4e. Weekly pattern picker (shared, New booking only)
- Seven day-of-week toggle chips (Mo–So), individually selectable, plus a
  single "Hours/day" input applied uniformly to all selected days.

5. NOTES (New AND Edit)
- Collapsed by default behind a "+ Add note" ghost button when empty;
  clicking expands the existing textarea in place. Pre-expanded if the
  booking already has note content (Edit).

6. VISUAL / STYLE
- Use the app's existing modal styling (white surface, spacing, purple
  accent) — this prompt changes structure and fields, not the overall
  visual language.
- Section labels: 12px, muted color, sentence case, no trailing punctuation.
- Badges and bar segments use the same neutral/warning/danger color
  conventions already used elsewhere in the app.

7. EDIT MODAL — COMPACT ROLE-BUDGET SUMMARY (Edit booking only)
Replace the current three stacked blocks ("This slot", "This slot — past vs.
future" with its explanatory paragraph and Release button, and the "Role
budget" table with its Budgeted/Logged/Invoiced/Re-plannable/Unplanned/Free
grid) and the two always-visible slot lists ("[Employee]'s slots on this
role" / "All employees on this role") with the following compact summary,
placed after Notes and before the Cancel/Delete/Save footer:

7a. Section header
- Small label "Role budget — [Role name]" with an (i) info icon to its
  right.
- Clicking/hovering the (i) icon opens a popover containing the FULL
  existing breakdown: Budgeted, Logged, Invoiced, Re-plannable, Unplanned,
  Free (2-column grid, same values/formatting as today) plus the existing
  explanatory note ("Budgeted = Invoiced + Re-plannable + Unplanned. Figures
  are live and match the Budget tab."). This is a direct relocation of the
  existing table into a tooltip — do not change the underlying values or
  formula.

7b. Budget bar
- A single horizontal segmented bar representing the role's Budgeted amount
  as its 0–100% scale (with the scale extending slightly past 100% if the
  role is over-committed):
  - Filled segment = Invoiced (solid accent color)
  - Adjacent segment = Re-plannable (lighter accent, bordered)
  - Any amount beyond the Budgeted threshold = Unplanned overshoot, shown
    as a distinct danger-colored sliver past a dashed vertical threshold
    line marking "Budgeted"
  - Axis labels below the bar: "0[unit]" / "[Budgeted amount] budgeted" /
    upper bound of the scale
- If there is undelivered past plan (planned days before today that were
  never logged), show a small clickable marker (clock icon) positioned on
  the bar at the relevant point. Clicking it opens a popover with:
  - The existing explanatory text ("Planned before today, never logged.
    Releasing frees this reservation — logged and future-planned work stays
    untouched.")
  - The existing "Release [X]d" action button, which performs the same
    release action as today.
  - This marker/popover only renders when there is a nonzero undelivered
    amount; when there is none, neither the marker nor the popover appears.

7c. Compact footer line
- Single muted 12px line below the bar: "[Bookable days] bookable days this
  slot → +[X]d" followed by a link-styled segment "[N] of [M] slots on this
  role · view all" (only shown if the employee/role combination has more
  than one slot; if this is the only slot, show just the bookable-days
  portion without the link).
- "View all" navigates to or opens the existing slot-list view (reuse
  whatever surfaces "[Employee]'s slots on this role" / "All employees on
  this role" today — this prompt does not require building a new view for
  that, just changing the modal's default display from always-expanded
  inline lists to an on-demand link).

7d. Data parity requirement
- All values shown in 7a–7c must be computed identically to the current
  implementation — this is a display-layer change only. Do not alter the
  underlying budget calculation, the release action's behavior, or the
  definitions of Invoiced/Re-plannable/Unplanned/Free.
