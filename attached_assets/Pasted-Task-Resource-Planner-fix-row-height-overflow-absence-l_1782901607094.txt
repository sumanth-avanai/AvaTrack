Task: Resource Planner — fix row height overflow, absence lane overlap, and
scroll-to-zoom not working (round 4)

IMPORTANT: Do not modify any Booking, Employee, Project, Holiday, or
Vacation data. This is a rendering/interaction fix only.

CONTEXT: This corrects two remaining issues after the previous three rounds:
- Row height is currently fixed/too small, so employees with multiple
  concurrent bookings (multiple "lanes" of bars stacked in one row) overflow
  visually into the next employee's row instead of the row growing to fit.
- Absence icons (holiday/vacation/sick/other) currently render overlapping
  booking bar labels instead of in their own clean lane.
- The scroll-to-zoom behavior added in the previous round does not work —
  scrolling the mouse wheel over the timeline grid does not change the zoom
  level.

1. DYNAMIC ROW HEIGHT BASED ON CONCURRENT LANES
- For each employee, determine the number of concurrent "lanes" needed in
  the visible date range: a lane is one horizontal track of non-overlapping
  booking segments. An employee with 3 simultaneous bookings across
  different projects/roles in the same period needs 3 lanes; an employee
  with only sequential (non-overlapping) bookings needs 1 lane.
- Each lane has a fixed height (e.g. 36px) with a small consistent gap
  between lanes (e.g. 4px).
- The employee row's total height = (number of lanes needed for that
  employee in the current visible range) x (lane height + gap), plus
  standard row padding. This must be computed per employee — do not use a
  single fixed row height for all employees.
- Row height must never cause visual overflow into the row above or below.
  Verify this specifically for employees with 3+ concurrent bookings (this
  was the exact failure observed: a 3-lane employee's content bled into
  the neighboring row).
- Absences (holiday/vacation/sick/other) occupy their own dedicated lane
  within the employee's row — they must NEVER render on top of / overlapping
  a booking bar's label or fill. If an employee has both a booking and an
  absence overlapping the same day (e.g. booked project + half-day sick),
  render them as separate lanes, not stacked/overlaid.
- Employees with no bookings and no absences in the visible range keep the
  existing single-lane "Available" row height (no change there).

2. FIX SCROLL-TO-ZOOM (functional bug, not a design change)
The mouse-wheel zoom feature specified in the previous round is not
working. Diagnose and fix, in particular checking for these common causes:
- The wheel event listener must be attached to the full timeline grid
  container (date headers + all employee rows), not just a sub-element like
  the header row alone.
- If the grid area also has its own horizontal-scroll container (from the
  "continuous scroll / focus window" behavior), make sure that container's
  own scroll handling does not swallow the wheel event before it reaches the
  zoom handler — the zoom handler should take priority for vertical wheel
  delta (deltaY) while horizontal panning (shift+wheel or trackpad
  horizontal scroll, deltaX) continues to scroll the timeline normally.
- The listener must call preventDefault() and be registered as
  { passive: false } — otherwise the browser will perform default scroll
  behavior instead of invoking the zoom handler.
- After fixing, verify scroll-to-zoom works when the cursor is anywhere
  over the grid (date headers, employee name column, and the day cells
  themselves), not just in one specific spot.

ACCEPTANCE CRITERIA
- [ ] An employee with 3 concurrent bookings gets a row exactly tall enough
      for 3 lanes, with no visual overlap into adjacent rows.
- [ ] An employee with 1 booking still gets a normal single-lane row height
      (no unnecessary extra whitespace).
- [ ] Absence icons always render in their own lane, never overlapping a
      booking bar's label or fill.
- [ ] Scrolling the mouse wheel while hovering anywhere over the timeline
      grid (headers, employee names, or day cells) changes the zoom level
      (Week → Month → Quarter → Year and back), clamped at both ends.
- [ ] Horizontal scrolling/panning of the timeline (from the continuous
      scroll behavior in the previous round) still works and is not broken
      by the zoom-on-wheel fix.
- [ ] Existing Bookings/Employees/Projects/Holidays/Vacations remain
      unchanged in the database.
